# Architecture

A collection of self-contained, browser-only utility tools served as static files. There is no backend, no build step for code, and no framework — each utility is plain HTML/CSS/JS that runs entirely in the user's browser.

## Top-level layout

```
utilities/
├── index.html              # Shell: nav bar + iframe content area
├── script.js               # Shell logic: navigate(), iframe param sync
├── style.css               # Shell-only styles (nav, settings overlay)
├── i18n.js                 # Translation runtime (shared)
├── settings.js             # Cookie-backed encrypted settings (shared)
├── index_lang.js           # Generated: shell translations
├── index_{en,ko,jp}.json   # Source: shell translations
├── build_lang.sh           # Regenerates *_lang.js from *_<lang>.json
├── unittest.html           # Test harness (loads source files + runs describe/it)
├── architecture.md         # This file
├── README.md
├── LICENSE
│
├── ColorPicker/            # ─┐
├── DayInfo/                #  │ Each subfolder is one self-contained
├── Dices/                  #  │ utility (HTML + CSS + JS + i18n).
├── JSONVisualizer/         #  │
├── QRGenerator/            #  │
└── Repayment/              # ─┘
```

Each utility folder follows the same convention:

```
<Module>/
├── <Module>.html           # Loads i18n.js, <Module>_lang.js, settings.js, then <Module>.js
├── <Module>.css
├── <Module>.js
├── <Module>_lang.js        # Generated by build_lang.sh
└── <Module>_{en,ko,jp}.json
```

Translation source-of-truth is the per-language JSON. `build_lang.sh` reads the JSONs and emits `<Module>_lang.js` (a single `I18N.register({...})` call).

## Application shell — `index.html` + `script.js`

The shell hosts each utility inside an `<iframe>`. It is **not** a SPA — each utility is its own document, navigated by swapping `iframe.src`.

- **Navigation**: `<a class="nav-link" data-src="QRGenerator/QRGenerator.html">` entries; `navigate(el)` updates `.active`, persists `lastpage` via `Settings.set`, swaps the iframe src.
- **Param propagation**: `iframeParams()` builds `?lang=X&theme=Y` and appends it to every iframe URL so the embedded utility starts in the right language and theme without having to read cookies itself.
- **Settings panel**: an overlay (`#settingsOverlay`) with three selects — language, default page, screen mode. Changes are saved to `utilities_setting` and re-applied immediately (the iframe reloads with the new params).

## Shared infrastructure

### `i18n.js` — translation runtime

```js
I18N.register({ en: {...}, ko: {...}, jp: {...} })
I18N.setLang('ko')                    // applies translations to the live DOM
I18N.t('key', fallback?, params?)     // dynamic lookup
```

Modules use these markup attributes:

| Attribute | Effect |
|---|---|
| `data-i18n` | Replaces `textContent` with the translation. |
| `data-i18n-placeholder` | Replaces input `placeholder`. |
| `data-i18n-title` | Replaces `title` attribute. |
| `data-i18n-aria` | Replaces `aria-label`. |

Dynamic strings (errors, announcements) use `I18N.t(key, fallback, {param: ...})`.

> Limitation: `I18N.register()` **overwrites** the registry, so it must be called once per page (the per-module `_lang.js` does this). The unit test harness loads all `_lang.js` files in sequence; this is fine because tests register their own data when needed.

### `settings.js` — encrypted cookie store

A single IIFE module exposing `Settings.{ready, get, set, save, applyScreenMode, readEncryptedCookie, writeEncryptedCookie}`.

Storage scheme:
1. Derive a per-device key: SHA-256 of the FingerprintJS `visitorId`, imported as an AES-GCM key via `window.crypto.subtle`.
2. Encrypt JSON: random 12-byte IV + AES-GCM ciphertext, base64-encoded into the cookie value.
3. Fallback (no `crypto.subtle` or no fingerprint): plain base64.

Two cookie shapes:

- `utilities_setting` — the shell's settings object (lang, defaultpage, lastpage, screenmode, plus per-module data like `repayment`).
- `utilities_qr_recent` and any other ad-hoc encrypted cookie written via `Settings.writeEncryptedCookie(name, jsonString)`. These reuse the same key/scheme but live in their own namespace.

All cookies are written with a **5-year `expires`** by `writeCookie()`, refreshed on every save (rolling expiry). Note: Chromium clamps cookie lifetime to 400 days regardless.

Boot sequence: `init()` → load FingerprintJS → derive key → read & decrypt cookie → apply URL `?lang` / `?theme` overrides → `I18N.setLang(...)` → `applyScreenMode(...)` → resolve `Settings.ready`. Modules typically gate first-render on `Settings.ready.then(...)` so they see the correct language and screen mode.

### Screen mode (light/dark)

`Settings.applyScreenMode(mode)` toggles `data-theme="light|dark"` on `<html>`. Each utility's CSS provides three variants: a base `:root` (light), `@media (prefers-color-scheme: dark)`, and explicit `:root[data-theme="dark"]` / `:root[data-theme="light"]` overrides so users can force a mode regardless of system preference.

### `build_lang.sh`

A node-driven shell script that walks a hard-coded list of module folders and writes `<Module>_lang.js` files combining the three per-language JSONs. Anyone adding a new utility must add an entry to this list.

## Utility modules

Each module is a single document. They share the same conventions (i18n attributes, Settings.ready gating, screen-mode-aware CSS) and differ in their core logic.

### ColorPicker
- **Purpose**: RGB color picker with name lookup against ~190 named colors (CSS standard + Korean/Japanese traditional palettes).
- **Core**: Three sliders + numeric inputs + a HEX text input, all bidirectionally synced. Computes complementary color, finds nearest named colors via weighted Euclidean distance.
- **Persistence**: none.
- **Notable**: All info sections (RGB/CMY/HSB) and the centered hex label are click-to-copy with a toast.

### DayInfo
- **Purpose**: Solar/Lunar calendar conversion with optional Korean public-holiday API.
- **Core**: Wraps everything in an IIFE. Two conversion paths — a precomputed `LUNAR_INFO` table (1900–2100) and an algorithmic fallback using astronomical computations (`calcNewMoon`, `solarLon`).
- **Persistence**: API service key via `Settings.set('dayInfo.apikey', ...)`.
- **Navigation**: `<<` / `<` / Today / `>` / `>>` buttons (week / day / today / day / week movement), plus epoch input.
- **Test exposure**: under `__UNITTEST__`, attaches pure helpers (`solarToLunar`, `gregorianToJD`, etc.) to `window._DayInfo`.

### Dices
- **Purpose**: Expected value + distribution table for arbitrary dice (sum or product).
- **Core**: Convolution over a `Map<value, BigInt count>` to get the exact distribution. Guards (`TABLE_LIMIT`, `COMPUTE_LIMIT`) abort runaway computations. Recalculation is debounced (200 ms) and assigned a monotonic `calcId` so older runs are discarded.
- **Persistence**: none.
- **Notable**: BigInt for exact combinatorics; sortable result table.

### JSONVisualizer
- **Purpose**: Radial mind-map for arbitrary JSON, with collapsible / draggable subtrees.
- **Core**: SVG edges (`<path>` quadratic Beziers) + absolutely-positioned HTML node divs. `layoutRadial()` recursively assigns angles weighted by leaf count. Drag transforms the subtree's CSS translation; canvas pan/zoom is tracked separately. Caps at `MAX_NODES = 500`.
- **Persistence**: none.
- **Notable**: Hand-rolled radial layout + interactive transforms; no charting library.

### QRGenerator
- **Purpose**: Generate a QR code from arbitrary text; keep recent entries.
- **Core**: `qrcode-generator@1.4.4` produces module data; we paint the canvas ourselves so the dark-mode fg/bg can swap. `detectMode()` chooses Numeric / Alphanumeric / Byte to maximize capacity (5596 / 3391 / 2331 chars at error level M).
- **Persistence**: `Settings.writeEncryptedCookie('utilities_qr_recent', ...)` stores up to 10 `{text, date}` entries. Items load on click; the textarea preserves the full text while the list shows a 60-char tail-truncated preview.
- **Accessibility**: `role="img"` + dynamic `aria-label` on the QR area, `aria-live` on char count and error region, `<button>`-based recent items (native keyboard support), Ctrl/Cmd+Enter shortcut to save, SR-only `#srStatus` live region for save/load announcements.
- **Test exposure**: IIFE-wrapped; under `__UNITTEST__`, exposes pure helpers (`detectMode`, `pad2`, `formatDate`, `truncateForDisplay`, `computeAddRecent`, `MODE_LIMITS`) on `window._QRGenerator` and returns before touching the DOM.

### Repayment
- **Purpose**: Loan amortization (equal-principal vs. annuity) with optional grace period.
- **Core**: IIFE module. `parseAmount()` strips non-digits; slider/input pairs stay synced. Computes per-period rows and renders a summary + amortization table.
- **Persistence**: Settings persistence under `repayment.{principal, period, holding, interest}`. Restored on `Settings.ready`.

## Cross-cutting conventions

- **No build step for code.** A user can serve the directory with any static server. Only translation `_lang.js` files are generated, by `build_lang.sh`.
- **External CDN deps** (loaded per-module via `<script>` tags): FingerprintJS (everywhere, for the encryption key), `qrcode-generator` (QRGenerator only). Everything else is hand-written.
- **IIFE wrapping** is used in `DayInfo.js`, `Repayment.js`, `QRGenerator.js`. New utilities should follow this to keep the global namespace clean.
- **Test exposure** uses the `window.__UNITTEST__` flag set by `unittest.html` *before* any source file loads. A module checks this flag and either exposes pure helpers on a `_<Module>` global or runs normal init.
- **Screen mode** support is mandatory: every module ships both `@media (prefers-color-scheme: dark)` and `[data-theme="dark"]` / `[data-theme="light"]` rules, sourcing colors from CSS variables.
- **i18n** is mandatory for visible text. Static strings use `data-i18n*` attributes; dynamic strings use `I18N.t(key, fallback, params)` so the fallback survives if the key is missing.

## Testing — `unittest.html`

A self-contained harness: hand-rolled `describe` / `it` / `expect` (`toBe`, `toEqual`, `toBeCloseTo`, `toBeTruthy`, `toBeNull`, `toBeGreaterThan`, `toBeLessThan`).

Mechanics:
1. Sets `window.__UNITTEST__ = true` in the first inline script so source files can detect test mode.
2. Stubs `document.getElementById` / `querySelector` to return a `fakeEl` for missing elements, and stubs `FingerprintJS` so `Settings` initializes without a real fingerprint.
3. Loads source files in dependency order, ending with the per-module `_lang.js` and module JS.
4. `Run Tests` button awaits `Settings.ready`, then iterates suites synchronously, recording pass/fail/elapsed.

Currently covered: I18N, Settings, DayInfo (date/calendar conversions, Julian Day, lunar table & algorithm), Dice (analytical EV), QRGenerator (mode detection, formatting, list dedup/cap).

## Adding a new utility

1. Create `<Module>/` with HTML, CSS, JS, and three `<Module>_<lang>.json` files.
2. The HTML loads `../i18n.js`, `<Module>_lang.js`, `../settings.js`, then `<Module>.js` — in that order.
3. Wrap the JS in an IIFE; gate first render on `Settings.ready`.
4. Add the module to `build_lang.sh`'s `pages` array.
5. Add a nav link to `index.html` (`data-src=...`) and a default-page option, plus `nav_<key>` and `page_<key>` translations in the three index JSONs and `index_lang.js`.
6. Use CSS variables and provide both prefers-color-scheme and `[data-theme]` overrides.
7. (Optional but recommended) Expose pure helpers under `window._<Module>` when `window.__UNITTEST__` is set, then add a suite to `unittest.html`.
